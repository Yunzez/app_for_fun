1. Overview & vision

Tally is a native habit tracker for iPhone, iPad, and Mac, replacing the user's daily-driver Awesome Habits. The app is designed for personal use first; App Store distribution is possible but not the primary goal.

The differentiating feature is per-habit task lists. The user uses habit timers as a container for either workouts or study, but the timer alone doesn't capture the structure of a session (today's lifts, today's reading queue). The app introduces:

  • A global Inbox — casual capture of any task, no commitment.
  • A per-habit task list — tasks scoped to that habit (e.g., "Workout" holds today's lift plan).
  • Drag & drop between Inbox and habits, and between habits.

2. Goals & non-goals

Goals

  • Match Awesome Habits feature parity for habits, timers, stats, widgets, and HealthKit ingestion.
  • Add the Inbox + per-habit tasks model, with drag/drop.
  • Sync via iCloud automatically across the user's devices, with no account.
  • Look and feel native on iOS and macOS; share ~95% of UI code via SwiftUI.
  • Make the color theme entirely abstract so palettes can be swapped later without touching views.

Non-goals (for now)

  • Multi-user collaboration / sharing.
  • Web app / non-Apple platforms.
  • Server-side anything. CloudKit is the only backend.
  • Apple Watch app in v1.0 — added in v1.1.
  • Generic plugin system / scripting.

4. Data model

SwiftData @Model entities. All entities have a UUID id and timestamps. Relationships use cascade delete from parent to children. Sketches below; final names and types may move during implementation.

4.1 Habit

@Model final class Habit {
    var id: UUID
    var name: String
    var iconName: String           // SF Symbol name or emoji
    var accentSlot: AccentSlot     // semantic palette slot, not raw hex
    var goalKind: GoalKind         // .count or .duration
    var goalTarget: Double         // count units, or seconds
    var schedule: HabitSchedule    // see 4.2
    var reminderTime: Date?        // time-of-day for daily nudge
    var healthBinding: HealthBinding?  // optional auto-fill source
    var sortOrder: Int
    var isArchived: Bool
    var createdAt: Date

    @Relationship(deleteRule: .cascade) var entries: [Entry] = []
    @Relationship(deleteRule: .cascade) var tasks: [TodoTask] = []
}

4.2 HabitSchedule (Codable struct stored on Habit)

enum HabitSchedule: Codable {
    case daily
    case weekly(weekdays: Set<Weekday>)        // e.g. [.mon, .wed, .fri]
    case monthly(daysOfMonth: Set<Int>)        // 1...31
    case flexible(timesPerWeek: Int)           // "3x / week, any days"
}

4.3 Entry — one per habit per day

@Model final class Entry {
    var id: UUID
    var date: Date         // day-aligned (start of local day)
    var value: Double      // count completed, or seconds completed
    var note: String?
    var createdAt: Date

    var habit: Habit?
    @Relationship(deleteRule: .cascade) var sessions: [TimerSession] = []
}

A duration habit's Entry.value is the sum of its sessions plus any manual additions. A count habit's Entry.value is the integer count.

4.4 TimerSession

@Model final class TimerSession {
    var id: UUID
    var startedAt: Date
    var endedAt: Date?
    var pausedAccumulated: TimeInterval   // total time spent paused
    var entry: Entry?
}

A session is "active" while endedAt == nil. Only one session may be active app-wide; this is enforced at the service layer, not the model.

4.5 TodoTask — Inbox or per-habit task

@Model final class TodoTask {
    var id: UUID
    var title: String
    var notes: String?
    var isDone: Bool
    var sortOrder: Int
    var dueDate: Date?               // optional, advisory only
    var createdAt: Date
    var completedAt: Date?

    // habit == nil  ⇒  Inbox task
    // habit != nil  ⇒  scoped to that habit
    var habit: Habit?
}

Tasks are persistent, not day-bound. A task lives in its container (Inbox or a specific habit) until it is marked done. There is no per-day duplication, no daily rollover logic. The user prunes the list by completing or deleting tasks. Recurring/scheduled tasks are out of scope for v1.0.

4.6 HealthBinding

enum HealthBinding: Codable {
    case steps                   // sum daily step count → Entry.value
    case activeEnergy            // kcal
    case workoutDuration         // seconds, any workout type
    case workoutType(HKWorkoutActivityType)
    case sleepHours              // hours asleep last night
    case mindfulMinutes
}

4.7 What is not in SwiftData

  • Themes & preferences: the active theme, week-start day, etc., live in UserDefaults via @AppStorage. Theme presets themselves are static Swift values (see §6).
  • Live timer state: the currently running timer is in-memory + persisted lightly to UserDefaults so it survives termination; it is materialized into a TimerSession when stopped.

4.8 Completion semantics

A habit is considered complete for a given day when both of the following hold:

  1. The day's Entry.value meets or exceeds Habit.goalTarget.
  2. The habit has no incomplete TodoTasks (vacuously true if the habit has no tasks).

Because tasks persist until done (§4.5), an undone task carried from a previous day still blocks "complete" status today — the user must finish it or remove it. This is the intended pressure: open tasks visibly delay the streak.

5. Sync & HealthKit

5.1 CloudKit

  • SwiftData container configured with the CloudKit option pointing at the app's private database.
  • No public/shared scope.
  • Conflict resolution: last-writer-wins (SwiftData default). Document any case where we override this.
  • All @Model properties must be CloudKit-compatible: nullable relationships, no required-non-optional refs, default values for new fields.

5.2 HealthKit

  • Read-only authorization.
  • Types: HKQuantityType for steps / active energy, HKWorkoutType, HKCategoryType.sleepAnalysis, HKCategoryType.mindfulSession.
  • A daily background HKObserverQuery updates today's Entry for habits with a HealthBinding.
  • Manual entries always win over HealthKit auto-fill (track via a source flag if needed).

6. Theme system

Theme is a value type with semantic slots only — no view ever names a hue. Concrete values can be re-tuned later without touching call sites.

6.1 Slots

6.2 Sketch

struct ThemePalette: Equatable {
    let id: String
    let displayName: String
    // semantic slots
    let accentPrimary: Color
    let accentSecondary: Color
    let backgroundPrimary: Color
    let backgroundSecondary: Color
    let backgroundTertiary: Color
    let textPrimary: Color
    let textSecondary: Color
    let textTertiary: Color
    let border: Color
    let success: Color
    let warning: Color
    let error: Color
    let habitPalette: [Color]   // exactly 8

    // Each slot resolves differently in light vs dark; ThemePalette.resolved(for:) returns the right one.
}

private struct ThemeKey: EnvironmentKey {
    static let defaultValue: ThemePalette = .slate
}

extension EnvironmentValues {
    var theme: ThemePalette {
        get { self[ThemeKey.self] }
        set { self[ThemeKey.self] = newValue }
    }
}

6.4 Rules of use

  • Views must read only from @Environment(\.theme). No Color.blue, no hex literals, no asset-catalog colors named after a hue ("BlueAccent"). Asset-catalog colors are fine if named semantically.
  • Per-habit accents pick an index (0...7) into theme.habitPalette, not a stored RGB.
  • System materials (.regularMaterial etc.) are allowed — they adapt to system appearance and don't fight the theme.

7. UX outline

iOS

  • Tab bar: Today (habits + their tasks for today), Inbox, Stats, Settings.
  • Tap a habit → detail sheet with timer, today's value, task list, notes, history.
  • Long-press a task → drag handle; drop on a habit row in Today, on the Inbox tab, or onto another habit.

macOS

  • NavigationSplitView: sidebar (Today / Inbox / each habit / Stats / Settings), detail pane.
  • Optional menu-bar item showing the active timer (v1.1).
  • Keyboard shortcuts for start/stop timer, new task, mark done.

Settings (shared)

  • Theme picker (6 presets, §6.3).
  • Week start day — user-configurable. Default reads from the system locale; the user can override (e.g., force Monday). Stats and heatmap respect this setting.
  • HealthKit permissions, notification permissions, iCloud sync status.

10. Resolved decisions

#QuestionDecision
1 App name Tally. Used as Xcode target, bundle id (TBD final reverse-DNS), and marketing name. Repo dir stays app_for_fun.
2 Task lifecycle Tasks persist in their container (Inbox or a habit) until marked done. No daily duplication, no rollover field — they simply stay until completed. (See §4.5.)
3 Habit completion A habit is complete for the day when goal value is met and there are no incomplete tasks attached to it. (See §4.8.)
4 Concurrent timers Single active timer app-wide. Starting a new one stops the previous. Enforced by TimerService.
5 Inbox → habit drag Move, not copy. A task has exactly one container at any time.
6 Week start day User-configurable. Default reads system locale; surfaced in Settings. Stats & heatmap respect the setting.
