1. How to read this plan

  • Sequenced by dependency, not calendar. Earlier milestones unblock later ones.
  • Each milestone ends with a concrete validation step the user runs on real devices.
  • Stop and dogfood whenever it's useful — most milestones are independently shippable to your own phone.
  • Two principles run through every milestone (see §14): no hardcoded colors, and verify CloudKit sync after any model change.

M0 Project bootstrap & sync smoke test

Stand up a multiplatform Xcode project and prove SwiftData + CloudKit round-trip works before any real domain code exists.

Deliverables
  • Xcode project Tally.xcodeproj with multiplatform Tally target (iOS 18 / macOS 15).
  • Bundle id, dev team set; signing works on user's Apple Developer account.
  • Capabilities: iCloud (CloudKit), HealthKit, Push (background notifications, needed for sync wake-ups), App Groups (for the future widget extension).
  • SwiftData container configured with CloudKit option, pointing at the private database.
  • One placeholder @Model (e.g., Ping with a date + label) and a one-screen view that adds a row.
  • Theme infrastructure stub: ThemePalette type, Environment key, one preset (Slate). All view code reads colors via the environment from day one.
  • Folder layout per design §9.
Validation
  • Add a Ping on iPhone → it appears on Mac within ~30s.
  • Delete on Mac → it disappears on iPhone.
  • Theme background applied via environment, not hex literals.
Risks
  • CloudKit signing/entitlement misconfiguration is the most common stall point — get past it here, not later.
  • If SwiftData behaves badly with sync at this stage, fall back to NSPersistentCloudKitContainer (decision logged in design §3).

M1 Data model & habit CRUD

Replace the placeholder model with the real schema and let the user create / edit / archive habits.

Deliverables
  • Habit, Entry, TimerSession, TodoTask, plus HabitSchedule, GoalKind, HealthBinding (per design §4).
  • HabitStore facade for create/update/archive/delete operations.
  • Habit list view (basic styling, sorted by sortOrder).
  • Create/edit habit form: name, icon (SF Symbol picker for now, emojis later), accent slot (one of habitPalette's 8), goal kind + target, schedule, optional reminder time, optional HealthBinding.
Validation
  • Create 3 habits with different goals + schedules on iPhone; all appear on Mac.
  • Edit a habit's name on Mac; change reflects on iPhone.
  • Archive a habit; it disappears from the active list on both platforms.
Risks
  • SwiftData's CloudKit constraints: every relationship must be optional, every property needs a default. Caught early by trying real data.
  • Schema migration: if anything changes after this milestone, plan a versioned migration before shipping to your own device.

M2 Today view, entries, single timer

Smallest useful loop: see today's habits, log progress, run a timer.

Deliverables
  • Today view: lists habits whose schedule includes today, with progress ring and current value vs goal.
  • Tap a count habit → +1 (and -1 / set value via long-press).
  • Tap a duration habit → start/stop timer.
  • TimerService enforcing single-active-session invariant; survives app backgrounding and termination (state in UserDefaults).
  • Manual entry edit (correct yesterday's value, etc.).
  • Habit detail view: today's value, history list, notes field on Entry.
Validation
  • Start a timer, lock the phone for 5 minutes, return — elapsed time accurate.
  • Start timer A, then start timer B → A stops automatically with its session saved.
  • Edit yesterday's Entry from the detail view; sync round-trips to Mac.
Risks
  • Background timer accuracy: don't try to keep a real timer running. Persist startedAt + paused intervals; compute elapsed on the fly.

M3 Tasks, Inbox, drag & drop

The differentiating feature. After this, the app is meaningfully better than Awesome Habits for the user.

Deliverables
  • Inbox tab + per-habit task list inside the habit detail view.
  • Task CRUD (title, optional notes, optional due date).
  • Drag tasks between Inbox ↔ habit, and habit ↔ habit, using SwiftUI .draggable/.dropDestination. Move semantics, not copy.
  • Apply design §10 #3 completion rule: a habit is complete only when goal value met and all attached tasks are done.
  • Today view shows the count of open tasks per habit.
Validation
  • Drop an Inbox task onto a habit row → task moves; Inbox no longer shows it.
  • Hit goal but leave one task open → habit shows "in progress", not complete.
  • Reorder tasks within a habit; order syncs to Mac.
Risks
  • Drag & drop on iOS vs macOS has small behavioral differences (long-press vs immediate). Test both.

M4 Stats, heatmap, streaks

Visualize accumulated data. By now you have weeks of real entries from dogfooding M2/M3.

Deliverables
  • Per-habit heatmap (12 weeks default; year view on iPad/Mac).
  • Current streak + longest streak (respecting flexible schedules — a "skipped, not failed" day breaks vs. tolerates depending on schedule kind).
  • Weekly & monthly summary (target hit rate, total time / count).
  • Stats tab aggregates across habits.
  • Settings → user-configurable week start day (default from system locale).
Validation
  • Back-date a few entries; heatmap reflects them on both platforms.
  • Change week start; heatmap and weekly summary realign.

M5 HealthKit auto-fill

Habits with a HealthBinding pull from HealthKit instead of needing manual entry.

Deliverables
  • NSHealthShareUsageDescription in Info.plist.
  • Read-only authorization flow during habit setup if a HealthBinding is selected.
  • HealthKitService running an HKObserverQuery per active binding; updates today's Entry in the background.
  • Manual entry overrides HealthKit value (track via a source flag if needed).
Validation
  • Create a "10k steps" habit bound to steps. Walk a bit. Today's value updates without opening the app (re-open and confirm).
  • Manually set today's value to 12000; HealthKit re-emit doesn't overwrite it.

M6 Reminders & notifications

Deliverables
  • Per-habit reminderTime → UNNotificationRequest scheduled with a calendar trigger that respects the habit's schedule.
  • Permission request on first habit-with-reminder.
  • NotificationService reschedules all reminders on app launch and after habit edits.
  • Optional: notification body shows progress ("Workout — 0 / 60 min today").
Validation
  • Set reminder to 1 minute from now on a daily habit; notification fires.
  • Edit reminder time; old notification gone, new one scheduled.

M7 Theme presets & per-habit accent

Six full themes. Theme infrastructure already exists from M0, so this is mostly tuning.

Deliverables
  • Six ThemePalette presets: Slate, Sky, Forest, Sunset, Lavender, Sand. Each with light + dark variants for every slot.
  • Per-preset 8-slot habitPalette for habit accent picker.
  • Settings → theme picker with live preview.
  • Audit pass: grep for Color., hex literals, and asset-catalog hue names; replace anything that slipped through.
Validation
  • Switch themes in Settings; entire UI (including widgets if M9 is done) updates without restart.
  • All six themes legible in light + dark, with WCAG AA-ish contrast on text and accents.

M8 macOS polish

Deliverables
  • NavigationSplitView sidebar (Today / Inbox / each habit / Stats / Settings).
  • Keyboard shortcuts: ⌘N new habit, ⌘T new task, Space start/stop timer on focused habit.
  • Window restoration; minimum window size; sensible default sizes.
  • iPadOS sanity pass (split view, regular vs compact size classes).
Validation
  • Use the Mac app for a full day without the phone; nothing feels like a literal iOS port.

M9 Widgets

Deliverables
  • TallyWidgets widget extension target.
  • App Group for shared SwiftData store access between app and widget.
  • Lock-screen widget: today's primary habit progress (rectangular + circular).
  • Home-screen widget: small/medium grid of habits with interactive complete buttons via AppIntent.
  • Widget timeline reload on habit/entry changes.
Validation
  • Tap a habit complete in a Home-screen widget; main app reflects it on next open and Mac sees it shortly after.
  • Lock-screen widget shows up-to-date progress without launching the app.

M10 Personal deploy & dogfood

Deliverables
  • Sign & install on user's iPhone, iPad, Mac via developer account (TestFlight optional).
  • Disable Awesome Habits notifications. Switch over to Tally.
  • Two weeks of daily use; log issues to docs/dogfood-log.html.
  • Decide: address surfaced bugs in a v1.0.x polish pass, or push toward v1.1 features (Watch, Live Activities, etc.).
Validation
  • You stop reflexively opening Awesome Habits.

14. Cross-cutting checks (run every milestone)

  • No hardcoded colors: grep -rn 'Color\.\|#[0-9a-fA-F]\{3,6\}' Tally/Views should turn up nothing meaningful. Per-habit accents reference theme.habitPalette[i], not raw hex.
  • Sync round-trip: after any @Model change, install on iPhone + Mac and verify create / edit / delete propagates. Watch for "schema mismatch" errors after migrations.
  • Both platforms: every UI feature gets at least a smoke test on iOS and macOS.
  • Privacy strings: any new system permission needs a usage description in Info.plist before the prompt fires.
  • Design doc currency: if a milestone changes a §4 model, update design.html in place.

15. Out of MVP scope (deferred to v1.1+)

  • Apple Watch app + complications
  • Live Activities + Dynamic Island timer
  • Vacation mode
  • CSV / JSON export & import
  • App Intents / Siri Shortcuts beyond what M9 needs
  • Apple Intelligence habit suggestions
  • macOS menu-bar timer
  • Recurring/scheduled tasks (today, tasks just persist)
  • Sharing / collaboration
