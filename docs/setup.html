Setup & M0 Validation

v0.1 · 2026-05-09 · companion to design.html and mvp-plan.html
This guide turns the source tree in this repo into a buildable Xcode project on your Mac, then walks through the M0 validation step (verify SwiftData + CloudKit sync between iPhone and Mac).

1. Prerequisites

  • macOS 15+ with Xcode 16 (Swift 6 toolchain).
  • Apple Developer account (for CloudKit, HealthKit, push entitlements).
  • XcodeGen — installed via Homebrew: brew install xcodegen. (We use this because hand-authored .xcodeproj files are noisy and the YAML config is reviewable.)

2. Materialize the Xcode project

From the repo root on your Mac:

cd app_for_fun
xcodegen
open Tally.xcodeproj

This generates Tally.xcodeproj from project.yml. The .xcodeproj is gitignored — regenerate any time the source tree changes.

3. Configure signing & identifiers

The project ships with placeholder identifiers. Update them once for your team.

3.1 Pick your bundle id

Decide on a reverse-DNS bundle id (e.g., com.yourname.Tally). Update three places:

  1. project.yml → options.bundleIdPrefix and targets.Tally.settings.base.PRODUCT_BUNDLE_IDENTIFIER.
  2. Tally/Resources/Tally.entitlements → com.apple.developer.icloud-container-identifiers (use iCloud.<your.bundle.id>) and com.apple.security.application-groups (use group.<your.bundle.id>).
  3. Re-run xcodegen after editing project.yml.

3.2 Set the development team in Xcode

Open Tally.xcodeproj → select the Tally target → Signing & Capabilities → choose your Team. Repeat for the macOS destination.

3.3 Create the iCloud container

In Xcode's Signing & Capabilities tab:

  1. Confirm the iCloud capability is present (it's wired in the entitlements file).
  2. Under Containers, click the + and create the container matching the id from §3.1 (Xcode auto-prefixes with iCloud.).
  3. Same for the App Group (group.<your.bundle.id>).
If signing fails: the most common cause is the iCloud container ID in the entitlements file not matching one that exists on your developer account. Either create it in the Apple Developer portal first, or let Xcode create it on your behalf via the + button.

4. Build & run

  • iPhone: select an iPhone destination (your physical device, not the simulator — CloudKit needs a signed-in iCloud account). Build and run.
  • Mac: select My Mac. Build and run.
  • Both devices must be signed into the same iCloud account.

5. M0 validation checklist

From mvp-plan.html M0. Tick all four to mark M0 done.

  1. Launch the app on iPhone. The list view says "No pings yet."
  2. Tap + three times. Three rows appear, newest first.
  3. Launch the app on the Mac. Within ~30 seconds, the same three rows appear.
  4. Swipe-delete a row on the Mac. It disappears from the iPhone within ~30 seconds.

If sync takes longer than ~60 seconds the first time, that's normal — CloudKit warm-up. Subsequent updates are usually within a few seconds.

6. Troubleshooting

  • Build error: "Provisioning profile doesn't include the iCloud container." The container id in entitlements doesn't match one on your account. Check §3.3.
  • Build error: "Use of unresolved identifier 'ModelConfiguration'." You're on a too-old Xcode. SwiftData needs Xcode 15+ and we target the iOS 18 SDK, so use Xcode 16.
  • App runs but rows don't sync. Both devices need to be signed into iCloud, and the iCloud account must have iCloud Drive enabled. Check Settings → [your name] → iCloud on each device.
  • SwiftData crash on launch about "missing default values." If you've added new model properties without defaults, CloudKit-backed SwiftData rejects them. Every property needs either a default value or be optional. (Already true for Ping; relevant when M1 lands.)

7. After M0

When the validation checklist passes, M0 is done. Proceed to M1 — data model & habit CRUD: the placeholder Ping model gets removed and the real Habit / Entry / TimerSession / TodoTask entities take its place.
